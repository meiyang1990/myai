## ADDED Requirements

### Requirement: 项目概要独立生成入口

系统 SHALL 提供独立的项目概要生成入口方法，接收用户提供的项目简要信息和项目本地根目录路径，结合自动扫描结果调用大模型生成项目概要（包含项目背景、架构设计等信息）。

#### Scenario: 用户提供项目简要信息生成概要

- **WHEN** 用户通过 `--generate-summary --project-info "这是一个电商平台后端服务"` 命令执行概要生成
- **THEN** 系统将用户提供的项目简要信息与自动采样的目录树、关键文件一起组装为 Prompt
- **AND** 调用大模型生成包含项目背景、架构设计、技术栈、模块职责等内容的概要文档
- **AND** 生成结果存储到长期记忆中

#### Scenario: 不提供项目简要信息时退回自动分析

- **WHEN** 用户未通过 `--project-info` 传入项目简要信息
- **THEN** 系统退回到纯自动扫描模式，仅依赖目录树和采样文件生成概要

### Requirement: 项目概要长期记忆存储

系统 SHALL 维护一个全局的长期记忆存储，建立「项目根目录 → 项目概要」的映射关系，供后续注释生成时作为 Prompt 上下文复用。

#### Scenario: 保存项目概要到长期记忆

- **WHEN** 项目概要成功生成
- **THEN** 系统将概要写入 `~/.code_comment_memory/project_summaries.json`
- **AND** 存储内容包含：项目路径、概要文本、用户提供的项目信息、生成时间戳

#### Scenario: 注释生成时优先加载长期记忆

- **WHEN** 执行注释生成流程
- **AND** 长期记忆中存在该项目的概要
- **AND** 未要求强制刷新
- **THEN** 直接从长期记忆加载概要，不再调用大模型重新分析

#### Scenario: 列出已记忆的项目概要

- **WHEN** 用户执行 `--list-memories` 命令
- **THEN** 系统列出所有已存储的项目路径、概要摘要和生成时间

#### Scenario: 删除指定项目的长期记忆

- **WHEN** 用户执行 `--remove-memory` 命令并指定项目路径
- **THEN** 系统从长期记忆中移除该项目的概要记录

## MODIFIED Requirements

### Requirement: 项目概要文档生成

系统 SHALL 扫描目标项目的源码文件，将具有代表性的文件内容提交给大模型，生成一份包含架构设计、技术栈、业务背景和模块职责的项目概要文档（Markdown 格式）。当用户提供了项目简要信息时，SHALL 将该信息作为额外上下文一起提交给大模型。

#### Scenario: 首次分析项目

- **WHEN** 用户首次对一个项目执行注释生成
- **AND** 该项目尚无缓存的概要文档，且长期记忆中也无该项目的概要
- **THEN** 系统自动挑选关键源码文件（入口文件、配置文件、核心模块等）
- **AND** 将这些文件内容组装为分析 prompt，调用大模型生成项目概要
- **AND** 概要文档包含：项目背景与目标、技术栈与框架、架构设计与分层、核心模块职责、模块间关系、关键设计模式
- **AND** 生成结果自动写入长期记忆

#### Scenario: 结合用户信息生成概要

- **WHEN** 用户通过 `--project-info` 提供了项目的简要信息
- **THEN** 系统在 Prompt 中增加「用户提供的项目简要信息」章节
- **AND** 大模型结合用户信息和自动扫描结果生成更准确的概要

#### Scenario: 控制分析成本

- **WHEN** 项目包含大量源码文件
- **THEN** 系统使用智能采样策略，优先选择入口文件、配置文件、包含关键字的核心文件
- **AND** 控制提交给大模型的总 token 数在合理范围内（可通过配置调整）

#### Scenario: 项目概要生成失败

- **WHEN** 大模型 API 调用失败或返回无效内容
- **THEN** 系统记录警告日志，继续执行后续注释生成（降级为无上下文模式）

### Requirement: 项目上下文持久化存储

系统 SHALL 使用基于文件系统的持久化方案，将项目概要文档存储在本地，避免重复调用大模型分析同一项目。持久化分为两层：项目内缓存（`.code_context/`）和全局长期记忆（`~/.code_comment_memory/`）。

#### Scenario: 存储概要文档到项目缓存

- **WHEN** 项目概要文档成功生成
- **THEN** 将概要文档以 JSON 格式存储到 `<项目根目录>/.code_context/project_summary.json`
- **AND** 存储内容包含：概要文本、生成时间戳、项目路径哈希、版本号

#### Scenario: 存储概要文档到全局长期记忆

- **WHEN** 项目概要文档成功生成
- **THEN** 同时将概要写入 `~/.code_comment_memory/project_summaries.json`
- **AND** 以项目路径 hash 为 key，后续可通过项目路径快速检索

#### Scenario: 加载已有概要（优先级）

- **WHEN** 再次对同一项目执行注释生成
- **THEN** 优先从全局长期记忆加载概要
- **AND** 如果长期记忆无记录，则从项目内缓存加载
- **AND** 如果均无记录，重新调用大模型生成

#### Scenario: 强制重新生成

- **WHEN** 用户通过命令行参数 `--refresh-context` 显式要求刷新
- **THEN** 忽略长期记忆和项目缓存，重新调用大模型生成项目概要并更新两处存储

#### Scenario: 缓存过期策略

- **WHEN** 缓存的概要文档超过可配置的有效期（默认 7 天）
- **THEN** 记录提示日志，建议用户使用 `--refresh-context` 更新
- **AND** 仍使用已有缓存（不自动重新生成，由用户决定）
