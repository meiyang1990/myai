## Context

当前系统在注释生成时仅向大模型提交单个文件的代码内容，缺乏项目全局信息。用户希望系统能够先理解项目整体架构和业务背景，再基于这个全局视角为每个文件生成更有深度的注释。

需要设计一个项目上下文分析与记忆机制，能够：
1. 智能采样项目关键文件并生成概要文档
2. 持久化存储概要文档，避免重复调用大模型
3. 在注释生成时将概要注入 prompt

## Goals / Non-Goals

- Goals:
  - 实现项目上下文分析模块（`project_context.py`）
  - 使用基于文件的持久化方案存储项目概要
  - 在注释生成 prompt 中注入项目上下文
  - 提供命令行参数控制上下文行为
  - 保持与现有流程的向后兼容

- Non-Goals:
  - 不使用向量数据库（项目概要文档较短，无需检索增强）
  - 不做增量更新（概要整体替换，不做部分修改）
  - 不做多模型支持（继续使用火山引擎大模型）

## Decisions

### Decision 1: 持久化方案选择 — 基于文件系统的 JSON 存储

**选择**：将项目概要以 JSON 文件形式存储在目标项目的 `.code_context/` 目录下。

**原因**：
- 项目概要文档是一段完整文本（通常 2000-5000 字），不需要向量化检索
- JSON 文件方案零依赖，不需要额外安装数据库
- 与 LangChain Memory 抽象解耦，简单直接
- 缓存天然跟随项目，换机器 clone 后也能保留（如果提交到 Git 的话）

**替代方案**：
- LangChain `ConversationBufferMemory` + 文件持久化：增加不必要的抽象层，概要非对话格式
- ChromaDB 向量存储：项目概要是整体文档而非需要语义检索的片段，向量化意义不大
- SQLite 存储：增加依赖，对于单一文档过重

### Decision 2: 智能文件采样策略

**选择**：基于规则的文件采样策略，优先选择：
1. **入口文件**：`main.*`、`app.*`、`index.*`、`Application.*`
2. **配置文件**：`pom.xml`、`package.json`、`build.gradle`、`requirements.txt`、`go.mod`
3. **README 文件**：`README.md`、`README`
4. **核心模块文件**：按目录分布采样，每个一级子目录选取代表性文件

**控制策略**：
- 单个文件截取前 200 行（避免超长文件占用过多 token）
- 总采样文件数上限可配置（默认 20 个）
- 总 token 预算可配置（默认 ~8000 字符，约 4000 token）

### Decision 3: Prompt 注入方式

**选择**：将项目概要追加到 System Prompt 中，作为 `## 项目背景信息` 节。

**原因**：
- System Prompt 是大模型的全局指令上下文，适合放置项目级信息
- 不影响现有 Human Prompt 的文件级变量
- 如果概要不可用，System Prompt 回退到原始版本（无缝降级）

### Decision 4: 缓存存储目录

**选择**：存储在 `<项目根目录>/.code_context/project_summary.json`

**原因**：
- 缓存跟随项目，不同项目间隔离
- `.code_context/` 名称清晰表达用途
- 建议用户将其加入 `.gitignore`（自动追加检查）

## Risks / Trade-offs

- **Token 消耗增加**：每次注释生成的 System Prompt 变长（约增加 2000-4000 token）→ 通过控制概要文档长度和 `MAX_TOKENS` 配置来缓解
- **首次运行耗时增加**：项目分析需额外调用一次大模型（约 10-30 秒）→ 通过缓存机制确保只在首次运行时产生
- **概要质量依赖采样质量**：如果采样的文件不具代表性，概要可能不准确 → 通过优先采样入口/配置文件来缓解
