## Context

当前系统通过 System Prompt 指导大模型生成中文注释，但缺少不可违反的硬性约束，导致大模型行为不够可控。同时，文件扫描阶段没有区分源码文件和测试代码文件，造成不必要的 API 调用浪费。

## Goals / Non-Goals

**Goals:**
- 在 System Prompt 层面引入硬性约束规则，确保大模型不修改源码、不为简单代码生成冗余注释
- 在文件扫描层面过滤单元测试文件和测试目录
- 规则以配置化方式管理，便于后续扩展

**Non-Goals:**
- 不做注释质量的自动化校验（如 AST 对比源码是否被修改）
- 不做测试文件注释的独立流程（当前直接跳过）
- 不修改现有注释生成的 LangChain 调用链结构

## Decisions

### 1. 硬性规则通过 System Prompt 文本约束实现

- **决策**: 在 `SYSTEM_PROMPT` 中新增「## 硬性规则（不可违反）」章节
- **原因**: 规则本质上是对大模型行为的约束，System Prompt 是最直接且零代码改动的方式
- **替代方案**: 通过后处理代码对比源码差异检测是否被修改 → 复杂度高，且无法覆盖语义等价但文本不同的修改

### 2. 测试文件过滤基于文件名模式匹配 + 目录名匹配

- **决策**: 使用 `fnmatch` 模式匹配测试文件名，同时将常见测试目录加入忽略列表
- **原因**: 主流语言的测试文件命名有明确的约定（如 `*Test.java`、`test_*.py`、`*.test.ts`），文件名匹配即可覆盖绝大多数场景
- **替代方案**: 解析文件内容检测测试框架注解/导入 → 需要读取文件内容后再判断，浪费 IO

### 3. 测试文件模式配置化

- **决策**: 在 `config.py` 中定义 `TEST_FILE_PATTERNS` 列表和 `DEFAULT_TEST_DIRS` 集合
- **原因**: 不同项目可能有不同的测试命名规范，配置化便于用户自定义扩展

## Risks / Trade-offs

- **风险**: System Prompt 的文本约束依赖大模型的遵守能力，无法 100% 保证 → 缓解：使用强硬措辞（MUST/NEVER）+ 低温度参数
- **风险**: 文件名匹配可能误判非测试文件（如 `TestUtils.java` 工具类） → 缓解：打印跳过提示供用户检查，后续可考虑加白名单机制
- **权衡**: 5 行以内函数的判定由大模型自行判断，可能有偏差 → 可接受，大模型对函数行数的判断准确度足够高
