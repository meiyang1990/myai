# Proposal: add-checkpoint-and-dedup

## Problem

当前系统存在以下三个问题：

1. **重复注释**：对已有中文注释的代码行或函数体，大模型可能再次添加中文注释导致注释重复叠加。
2. **文件级无断点恢复**：每次重跑程序都会从头处理所有文件，即使某些文件已经成功生成过中文注释，也会重复调用大模型，浪费时间和 API 调用量。
3. **目录级无断点恢复**：当某个目录下所有文件和子目录已全部处理完毕时，下次重跑仍会重新进入该目录扫描处理。

## Solution

### 1. 避免重复注释（Prompt 层面）

在 `SYSTEM_PROMPT` 的硬性规则中新增第 6 条规则：
- 对于已经包含中文注释的代码行，不要重复添加中文注释
- 对于函数/方法上方已有中文文档注释的，不要再生成重复的函数级注释
- 识别已有注释并保留，仅对尚无中文注释的代码段落添加新注释

### 2. 文件级断点恢复

新增 `progress_tracker.py` 模块：
- 在目标项目的 `.code_context/` 目录下维护 `progress.json` 文件
- 记录每个已成功处理的文件的相对路径和处理时间戳
- `do_generate()` 处理文件前先查询进度记录，已处理的文件直接跳过
- 提供 `--reset-progress` 命令行参数清除进度记录

### 3. 目录级断点恢复

在 `progress_tracker.py` 中扩展目录级记录：
- 当某个目录下所有文件和子目录已全部被处理过时，自动标记该目录为"已完成"
- `_scan_directory()` 递归时先检查目录是否已完成，已完成则跳过
- 目录完成状态由文件完成状态自底向上推导

## Affected Specs

- `source-code-reader`: 新增断点检查集成，扫描时可跳过已完成的目录
- `comment-generator`: 新增避免重复注释的硬性规则，新增进度跟踪集成

## Risks

- 如果文件内容在两次运行之间发生了变化，断点恢复可能跳过需要重新处理的文件。后续可通过文件 hash 校验增强，但本次暂不实现。
